§1

TL;DR

Lecture éclair en 4 cartes — chacune un angle distinct du projet.

Livré

App React/TS responsive : recherche livres, fiche détaillée, bibliothèque utilisateur, auth, carrousels livres liés. Déployée Vercel.

Process

Figma → GitHub Project → branches feature → PR conventionnelles → Husky/Commitlint → Vercel. 20+ PR mergées en 10 jours.

Arbitrage clé

API Open Library via proxy Express local (port 3001) plutôt que Google Books — meilleure couverture FR + modèle works/editions adapté aux livres liés.

Hors scope

Persistance (bookmarks, lu/non-lu, réservations, cloche) non implémentée — pas de backend prêt → mocks + login différé, documentés en issues phase:logic.

§6

Points forts techniques

Six choix structurants. Premier ouvert pour le point clé — cliquer les autres pour les explorer.

a11y Infra Drawer accessible réutilisable

Drawer Radix/Vaul intégré avec focus trap automatique + attribut inert sur le reste du DOM. Hook useInertWhileDrawerOpen posé en PR #53 (US2 réservation), réutilisable pour bookmark / sélection liste. Une seule infra a11y, plusieurs use-cases.

TypeScript Types discriminés pour CTA contextuel

BookCTA a 6 variantes selon l'état du livre (disponible, réservé par moi, réservé par autre, emprunté par moi, emprunté par autre, indisponible). Type discriminé sur clé status → exhaustivité vérifiée par le compilateur, pas de branche oubliée à l'exécution.

refactor Slugs en URL pour étagères

PR #51 : passage de clés internes à des slugs URL-safe pour /library/shelf/:slug. Séparation types ↔ helpers ↔ data. Plus simple à partager, plus lisible, indépendant de l'ID interne.

DX Sandbox dev pour itération UI rapide

src/pages/dev/BookCTASandbox.tsx rend toutes les variantes CTA côte-à-côte. Débuggage / validation visuels sans cliquer dans l'app. Pattern réutilisable pour autres composants à variants.

structure Architecture features / pages / shared

Séparation domaine (features/book, features/library) vs routage (pages/) vs primitives partagées (shared/ui). Chaque feature porte ses composants UI, types, hooks.

qualité Conventional Commits + Husky verrouillés

Pre-commit hook : Prettier + ESLint sur fichiers modifiés (lint-staged). Commit-msg hook : Commitlint valide type(scope): sujet. Aucun bypass (--no-verify) dans l'historique. Log git lisible bout-en-bout.

§7

Arbitrages & limites assumées

Ce qu'on a choisi de ne pas faire — et pourquoi.

Open Library plutôt que Google Books

arbitrage API

Brief demandait Google Books. Choix d'Open Library : sans clé d'API, couverture FR meilleure, modèle works/editions plus naturel pour livres liés. Dépendance @googleapis/books conservée pour fallback éventuel.

Persistance hors scope

hors scope

Bookmarks, lu/non-lu, notification cloche, réservation calendrier → UI prête, fetch non câblé. Issues phase:logic ouvertes assument la dette (#31, #33, #35, #37, #39, #41, #43, #47).

Auth UI sans backend

UI seul

Login / Register fonctionnels visuellement, état isLoggedIn local. Pas de JWT, pas de session serveur. Cohérent avec absence de backend.

Revert background wooden shelf

qualité merge

PR #60 mergée puis revertée (#61) côté main car régression UI desktop. Re-travaillée en draft #63. Illustre la discipline merge/revert via PR plutôt que force-push.
