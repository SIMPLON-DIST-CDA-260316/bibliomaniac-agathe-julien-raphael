À lire en 90 secondes — les 5 idées-clé
  1. Un hook a une intention, et son intention dit où il vit. Les dossiers hooks/ et types/ disparaissent (FSD les nomme « bad segment names »). Fetch → api/. Logique métier → model/. Helper pur sans state → fonction (pas hook) dans lib/. Types → model/<entity>.types.ts.
  2. Une slice expose ses composites, pas ses atomes. Le public API features/<slice>/index.ts est minimaliste : composites haut-niveau + hooks + types + utilities. Les atomes UI (BookCover, BookMeta…) restent privés. Si tu veux en importer un depuis l'extérieur, c'est qu'il manque un composite — ouvre une discussion plutôt que d'éroder le public API.
  3. Une source externe ne contamine qu'un seul fichier. OpenLibrary (et toute API tierce future) vit dans api/mappers/<source>.ts — c'est le seul endroit autorisé à nommer la source. Le modèle Book reste stable, l'app ignore d'où viennent les données. Pattern anti-corruption layer.
  4. Tous les hooks data-fetching exposent { data, isLoading, error }, même quand la donnée est synchrone (mock). Convention TanStack Query / SWR. Le jour où un vrai fetch arrive, les consumers ne bougent pas — c'est le pattern « boundary-swap ».
  5. Les règles d'import FSD sont enforcées mécaniquement. eslint-plugin-fsd-lint bloque au commit (via husky/lint-staged) les imports cross-slice directs et les sidesteps de public API. Tu n'as plus à mémoriser les règles : tu apprends en réagissant aux messages d'erreur.
§0 Vocabulaire FSD

Layer, slice, segment — 3 mots à savoir avant tout le reste.

FSD organise le code en 3 niveaux d'imbrication. Tout le refactor agit au niveau le plus profond (segments). Comprendre cette hiérarchie débloque l'ensemble du document.

niveau 1 Layer

Niveau d'abstraction. FSD en définit 6 prédéfinis (on ne crée pas les siens) : app, pages, widgets, features, entities, shared.

Règle clé : les imports descendent uniquement. features/ peut importer shared/, jamais l'inverse.
Dans notre projet : app, pages, features, shared (pas d'entities ni widgets).
niveau 2 Slice

Découpage métier à l'intérieur d'un layer. Tu en crées autant que tu veux. Chaque slice représente un domaine isolé.

Règle clé : deux slices d'un même layer ne peuvent pas s'importer directement (isolation horizontale, enforcée par fsd/no-cross-slice-dependency).
Dans notre projet : features/book, features/navigation. Note : app et shared n'ont pas de slices.
niveau 3 Segment

Découpage technique à l'intérieur d'une slice (ou d'un layer sans slices). Décrit l'intention du code : où il interagit, ce qu'il fait.

Règle clé : les noms standards sont api, model, lib, ui, config. Bannis : hooks, types, components.
Dans notre projet : tout le refactor déplace des fichiers entre segments sans toucher aux layers ni aux slices.
🎯
À retenir pour lire le reste du document
  • Layer ⊃ Slice ⊃ Segment. Chemin complet : {layer}/{slice}/{segment}/fichier.ts — ex : features/book/api/useBookDetail.ts.
  • Le layer app n'a pas de slices — il a directement des segments. C'est la subtilité derrière l'axe 3 : on a ajouté les segments routes/ et styles/ dans app/.
  • Ce refactor n'a créé ni layer ni slice. Les 9 hooks de features/book ont été reclassés entre les segments api/, model/, lib/, mocks/ selon leur intention.

§3 Les 5 axes du refactor

Cinq décisions, cinq règles.

Chaque axe est isolé. Tu peux lire celui qui te concerne et ignorer les autres.

Axe 1

Hooks et types reclassés dans des segments idiomatiques

Citation officielle FSD : « components, hooks, and types are bad segment names because they aren't that helpful when you're looking for code. »

api/

Data-fetching, request functions, data types, mappers.

useBookDetail · useBookSearch · useUserLibrary…
model/

Schémas, interfaces, stores, business logic.

book.types · useDiscoveryBooks · useRelatedBooks
lib/

Helpers internes purs (pas de state, pas de side-effect).

filterBooksByQuery · discoveryCache · categories
mocks/

Fixtures statiques pour développer sans backend.

MOCK_USER_LIBRARY · MOCK_USER_SHELVES…
› Pourquoi useBookSearch est devenu une fonction pure ?

L'ancien useBookSearch faisait useMemo(() => books.filter(...), [books, query]). Opération cheap (substring), pas de side-effect, pas de state. Le useMemo est un détail de performance du caller, pas une responsabilité du hook.

Convertir en fonction pure : plus de règle des hooks, utilisable hors composant React. Les callers appliquent useMemo eux-mêmes quand pertinent.

› Pourquoi useSampleBooks a été supprimé ?

useSampleBooks faisait useMemo(() => SAMPLE_BOOKS, []) autour d'une constante top-level. La constante est déjà stable → useMemo no-op. Le hook ne fait rien.

Le refactor sépare deux préoccupations qui étaient mélangées :

  1. Fixture statique : MOCK_USER_LIBRARY dans mocks/userLibrary.mock.ts.
  2. Contrat data-fetching : useUserLibrary() dans api/, qui retourne { data, isLoading, error }.

C'est le pattern « boundary-swap » : on figure dès aujourd'hui la shape asynchrone réelle, même quand la data est synchrone. Demain on branche un vrai fetch, les composants ne bougent pas.

Axe 2

Anti-corruption layer pour les sources externes

étape 1 — le problème

Le même mapper, écrit deux fois (avant ce refactor)

Deux hooks du dossier features/book/hooks/ faisaient appel à OpenLibrary : useOpenLibraryBooks et useSearchBooks. Chacun redéfinissait en interne l'interface du Doc OpenLibrary et la fonction de conversion vers Book. 

// features/book/hooks/useOpenLibraryBooks.ts
interface OpenLibraryBook { /* version A */ author_name?: string[]; cover_i?: number; key: string; first_publish_year?: number }
function mapToBook(d: OpenLibraryBook): Book | null { /* logique de mapping */ }

// features/book/hooks/useSearchBooks.ts
interface OpenLibraryBook { /* version B */ author_name?: string[]; cover_i?: number; key: string; first_publish_year?: number; isbn?: string[] }
function mapToBook(d: OpenLibraryBook): Book | null { /* logique de mapping (presque) identique */ }

Différence subtile : le second a un champ isbn en plus, jamais consommé. Mais surtout : si OpenLibrary change son schéma, il faut le corriger à deux endroits en restant cohérent — divergence silencieuse garantie.

étape 2 — la solution

Un nouveau fichier dédié, posé au commit c31c7f8

Le refactor crée features/book/api/mappers/openLibrary.ts — fichier qui n'existait pas avant. Il devient le seul endroit du projet autorisé à nommer OpenLibrary. Il exporte trois choses : 

// features/book/api/mappers/openLibrary.ts — NOUVEAU FICHIER

export interface OpenLibraryDoc {
  author_name?: string[]
  cover_i?: number
  key: string
  first_publish_year?: number
  // ...shape brut OpenLibrary (snake_case, optionnels)
}

export interface OpenLibraryResponse {
  docs: OpenLibraryDoc[]
  // enveloppe HTTP
}

export function mapOpenLibraryToBook(doc: OpenLibraryDoc): Book | null {
  if (!doc.cover_i) return null            // règle métier UI : pas de livre sans couverture
  return {
    id: doc.key,
    title: doc.title ?? '',
    author: doc.author_name?.[0] ?? '',
    coverUrl: `https://covers.openlibrary.org/b/id/${doc.cover_i}-L.jpg`,
    publishedDate: doc.first_publish_year ?? null,
    // tous les champs manquants comblés en null
    genre: null, language: null, pageCount: null, rating: null,
    copies: null, summary: null, series: null, seriesOrder: null,
    collection: null, collectionOrder: null,
  }
}

Les deux hooks (renommés en useDiscoveryCategoryBooks et useBookSearch) importent ce mapper au lieu de redéfinir le leur. Plus de duplication, source de vérité unique.

étape 3 — le pattern

Anti-corruption layer (ACL)

Pattern d'architecture : isoler le format externe (snake_case OpenLibrary, schéma Google Books, REST de ton backend prod…) du modèle interne stable. Trois rôles, trois fichiers, une seule traversée autorisée :

externe
OpenLibraryDoc

Shape brut côté OpenLibrary. author_name, first_publish_year, cover_i. snake_case, champs optionnels. Leur schéma, pas le nôtre.

conversion
mapOpenLibraryToBook(doc) → Book | null

Filtre (pas de cover → null), renomme (cover_i → coverUrl), comble en null les champs absents. Seul endroit qui « parle leur langage ».

interne
Book

Modèle stable du projet. id, title, author, coverUrl… Connu de toute l'app, ignore d'où viennent les données.

Conséquence — règle pour une nouvelle source externe (Google Books, ISBN-DB…)

  1. Créer features/book/api/mappers/googleBooks.ts avec interface GoogleBooksVolume, interface GoogleBooksResponse et fonction mapGoogleBooksToBook(volume) → Book | null.
  2. Le nouveau hook (api/useBookSummary.ts par exemple) importe uniquement ce mapper.
  3. Aucun autre fichier ne change. Modèle Book figé, composants n'apprennent rien sur Google Books. La diff dans la PR est strictement contenue à api/.

Contrat unifié des hooks data-fetching

interface QueryResult<T> {
  data: T
  isLoading: boolean
  error: Error | null
}

Convention TanStack Query / SWR. Si on adopte une de ces libs plus tard, les composants n'ont rien à toucher. Pourquoi data (générique) plutôt que books / shelves (domaine) ? Le shape n'est pas couplé au nom du domaine — si demain le hook renvoie un résultat paginé, le nom data reste valide. Côté caller, on rename à la destructuration quand besoin de clarté locale : const { data: books } = useUserLibrary().

Axe 5

ESLint plugin pour enforcer les règles FSD

eslint-plugin-fsd-lint@1.2.1 installé et configuré dans eslint.config.js. Six règles actives. Le plugin a détecté 47 violations à l'install — toutes résolues, 0 à la fin.

Règle Sévérité Ce qu'elle bloque
fsd/forbidden-importserrorImport qui viole la direction des layers (ex : entities importe features)
fsd/no-cross-slice-dependencyerrorImport direct d'une slice depuis une autre slice du même layer
fsd/no-public-api-sidesteperrorImport direct vers un sous-fichier (@/features/book/ui/Foo) au lieu du index.ts

fsd/ordered-importswarningOrdre des imports (auto-fix)

philosophie Chaque règle protège un invariant d'architecture

ESLint n'est pas un garde-fou aléatoire. Chaque règle FSD correspond à une garantie que l'architecture promet — si la règle saute, la garantie aussi. Comprendre la garantie = ne plus se prendre un mur quand tu écris du nouveau code.

  • no-public-api-sidestep protège la liberté de refactor interne d'une slice. Le mainteneur peut réorganiser ui/, model/, lib/ sans casser les pages, tant qu'il préserve son index.ts.
  • no-cross-slice-dependency protège l'isolation métier. Une slice doit pouvoir évoluer (ou être supprimée) sans casser les autres slices du même layer.
  • forbidden-imports protège la direction des imports (descendante uniquement). Sans ça, on retombe en spaghetti circulaire.
  • no-ui-in-business-logic protège la séparation UI / logique. Le model/ et l'api/ doivent rester testables sans React.

cas concrets Trois des 47 violations résolues sur cette branche

› Cas 1 — no-public-api-sidestep (le plus fréquent)

Symptôme — une page importait directement un composant interne :

// ❌ BookDetailPage.tsx (avant)
import { BookCover } from '@/features/book/ui/BookCover'
import { BookCTA }   from '@/features/book/ui/BookCTA'
import { BookMeta }  from '@/features/book/ui/BookMeta'
// ...6 imports vers des fichiers internes de la slice

Erreur ESLint :

error  Direct import from '@/features/book/ui/BookCover' is not allowed.
       Use the public API (index file) instead.   fsd/no-public-api-sidestep

Mauvais fix réflexe : ajouter BookCover dans features/book/index.ts pour rendre l'erreur silencieuse. Ça résout la règle mais érode le public API — on expose un atome qui devrait rester interne.

Bon fix : créer un composite haut-niveau (BookDetail) dans la slice, qui assemble les atomes en interne. Exposer le composite. La page consomme un seul item.

// ✅ BookDetailPage.tsx (après)
import { BookDetail } from '@/features/book'
<BookDetail book={book} leadingSlot={...} />
› Cas 2 — no-relative-imports cross-slice

Symptôme — un fichier importait un truc d'une autre slice via chemin relatif :

// ❌ features/book/ui/BookCard.tsx
import { NavLink } from '../../navigation/model/navigation.types'

Erreur ESLint :

error  Relative imports outside the current slice are not allowed.
       fsd/no-relative-imports

Pourquoi cette règle : un chemin relatif cross-slice contourne le public API (on peut atterrir directement dans model/ d'une autre slice). Plus subtil : ça casse en silence si la slice cible est déplacée.

Config locale : allowSameSlice: true permet les imports relatifs à l'intérieur d'une slice (entre segments). ignoreImportPatterns: ['\\.css$'] permet les imports CSS relatifs (cas particulier : pas de public API pour du CSS).

Fix : passer par le public API absolu.

// ✅ features/book/ui/BookCard.tsx
// (mais en vrai, ce cas ne devrait pas exister : book ne devrait pas
//  dépendre de navigation. cf. cas 3 ci-dessous)
› Cas 3 — no-cross-slice-dependency

Symptôme — une slice tente d'importer une autre slice du même layer :

// ❌ features/book/model/useDiscoveryBooks.ts
import type { NavLink } from '@/features/navigation'

Erreur ESLint :

error  Slice 'book' cannot import from slice 'navigation'.
       fsd/no-cross-slice-dependency

Pourquoi cette règle : deux slices d'un même layer doivent rester orthogonales. Si book dépend de navigation, supprimer navigation casse book.

Fix possibles :

  • Si le type est vraiment partagé par les deux slices → le déplacer vers shared/ (layer transversal accessible à tous).
  • Si le type appartient à navigation et que book en avait besoin par erreur → revoir la conception, c'est probablement navigation qui doit appeler book (ou un composant pages/ qui orchestre les deux).
› Quand un PR échoue — workflow

Husky/lint-staged exécute eslint --fix sur les fichiers stagés. Si une règle FSD est violée :

error  Direct import from '@/features/book/ui/BookCover' is not allowed.
       Use the public API (index file) instead.   fsd/no-public-api-sidestep

→ Lis l'erreur. Identifie la règle. Si tu te retrouves à devoir exposer un atome dans index.ts parce qu'une page en a besoin, pause : c'est probablement le signe qu'il manque un composite haut-niveau. Ouvre une discussion plutôt que d'éroder le public API.
